Create at https://www.dropbox.com/developers/apps → "Create app" → Scoped Access → Full Dropbox OR App folder (either works for the `shared_link` parameter) → Permissions tab → check `files.metadata.read` → Submit → Settings tab → Generated access token → click "Generate". The token starts with `sl.` and is valid for ~4 hours unless refreshed. (Adapted from `.planning/spikes/001-dropbox-photo-audit/list-dropbox.mjs:5-9`.)

    4. `## Steps` (numbered):

       1. **Create the Dropbox app + token** (per Before You Start). Export it for the current shell:
          `export DROPBOX_TOKEN='sl.…'`
          The token is sensitive — never commit it, log it, or paste it into chat. Phase 26's script redacts it from every error message via the pattern at `scripts/upload-plates.js:112`.

       2. **Verify the script loads** (dry-run, no real network call required):
          `DRY_RUN=1 DROPBOX_TOKEN=sl.… npm run photos:ingest`
          Expected: prints the first 5 entries (path, size, content_hash) from the Dropbox folder and exits 0 without writing the manifest. If you see a Dropbox 401, the token is wrong or expired.

       3. **Run the full ingest in tmux**:
          `tmux new -s ingest`
          `DROPBOX_TOKEN=sl.… npm run photos:ingest`
          Detach with `Ctrl-b d`. Reattach later with `tmux attach -t ingest`. The job runs for ~5–15 minutes against the audit corpus (5,000 files; only metadata, no bytes), longer if Dropbox rate-limits.

       4. **What to expect in the log output** (per-stage lines, OPS-01):
          Each per-file line has the format `<ISO timestamp> <content_hash prefix> <action> <outcome> [extra]` — for example `2026-05-22T12:34:56.789Z abc123def456 classify         clean-match  abagrotis apposita`. Transient errors emit `[ingest-photos] transient error on <label> (attempt N/5) — retrying in Ns`. The final summary line reports counts per match_bucket.

       5. **Interpret the manifest columns** — open `data/species-photos-manifest.csv` in a spreadsheet. Schema (D-05):

          | Column | Meaning |
          |---|---|
          | content_hash | Dropbox's deterministic hash; row identity (D-04) |
          | dropbox_path | Full path inside the shared link |
          | size_bytes | File size in bytes |
          | server_modified | ISO timestamp from Dropbox |
          | filename_raw | The exact filename from Dropbox |
          | binomial_raw | What the parser extracted (e.g. `abagrotis apposita`) |
          | specimen_id | `A`/`B`/… OR `OSAC_…` / `WWUC*` |
          | view | `D` (dorsal) or `V` (ventral) |
          | binomial_resolved | Matched species binomial (empty unless clean-match) |
          | species_slug | Matched species slug (empty unless clean-match) |
          | match_bucket | `clean-match` / `genus-only` / `likely-synonym` / `provisional` / `unparseable` |
          | status | `discovered` from Phase 26 (Phases 28+29 advance it) |
          | last_error | Empty on success; populated for retry-exhausted Dropbox errors only |

       6. **Resume after interruption** — kill the tmux session, restart it with the SAME command. The script reads the existing manifest, builds a Set of seen content_hashes (D-04), and skips any file already in the manifest (INGEST-05, OPS-03). Each skipped file logs one line `<timestamp> <hash> skip already-in-manifest`.

       7. **Re-sort the manifest for curator review** (D-12): `npm run photos:investigate`. This re-runs the script in RESORT_ONLY=1 mode — no Dropbox calls, just reads the manifest, sorts so the `genus-only` / `likely-synonym` / `provisional` / `unparseable` rows come first (ordered by binomial frequency, matching the spike's top-unmatched ordering: `Smerinthus ophthalmica`, `Grammia *`, `Eupithecia *`), and writes it back. Open the file in a spreadsheet and work top-down. Phase 27 will use the top rows to seed `data/species-synonyms.csv`.

       8. **The `*custom/` Dropbox subfolder is deliberately skipped** (D-11). The non-recursive listing emits a single `folder` entry for it which the script logs as `folder-skip` and ignores. Inspecting `*custom/` is a deferred task tracked in `.planning/STATE.md`; do not enable recursive listing without a separate planning round.

    6. `## When Things Go Wrong`:
       - **Dropbox 401 / 403**: token expired or wrong scope. Regenerate with `files.metadata.read` scope.
       - **Dropbox 429** (rate limit): script retries up to 5 times with 2s/4s/8s/16s/32s backoff (D-15). If still failing, the page errors out and the partial manifest is saved; rerun later.
       - **`recursive: true` not allowed**: should never happen — the script hardcodes `recursive: false` per skill reference §"Constraints". If you see this error, something has been edited.
       - **Token leaked in an error message**: should never happen — `scripts/ingest-photos.js` follows the redaction pattern at `scripts/upload-plates.js:112`. If you see a literal `sl.…` substring in an error, file a bug (the redaction broke).

    - Any rclone / FTP setup (that's UPLOADING_IMAGES.md territory for the LFS-era workflow; Phase 26 is API-only).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator hands → Dropbox API | Live token in shell history; mitigated by `export DROPBOX_TOKEN=...` (not persisted via `.env`). |
| Dropbox API → committed manifest | Real Dropbox filenames now flow into a committed file; CSV-quoting is locked by Plan 02's writeManifest. |
| Manifest in git → public history | The manifest is committed to the public repo; it contains filenames and content_hashes but no secrets. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26.04-01 | Information disclosure | DROPBOX_TOKEN in shell history | accept | The operator is responsible for token hygiene. The runbook recommends `export DROPBOX_TOKEN='...'` in an interactive shell rather than persisting it. Plan 03's redaction handles the in-process surface; shell history is out of scope for the script. |
| T-26.04-02 | Information disclosure | manifest committed with content_hash + filenames | accept | content_hash and filename are not sensitive — they appear in the Dropbox URL's listing for anyone with the share link. No PII, no secrets. The CONTEXT.md explicitly chose committed-CSV (D-01) for curator visibility. |
| T-26.04-03 | Tampering | operator runs malicious modified script | accept | Out of threat model — the operator is trusted (the project owner). Any local modification of `scripts/ingest-photos.js` is a project-internal concern, not a Phase 26 threat. |
| T-26.04-04 | Repudiation | bucket distribution does not match audit | mitigate | The human-verify checkpoint explicitly compares the produced distribution against the audit's known-good numbers. A divergence > ±3% triggers diagnosis against Plan 03 (likely a parser or cascade regression). The acceptance check is mechanical (awk on the manifest column) so the human is not relied on for arithmetic. |
| T-26.04-SC | Tampering | No new package installs in this plan | accept | This plan adds zero npm dependencies. Verified via the Plan 03 allowlist check. |
