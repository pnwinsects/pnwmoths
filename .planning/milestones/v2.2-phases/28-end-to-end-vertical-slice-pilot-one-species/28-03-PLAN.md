Phase 13 / Phase 18 HTTP PUT pattern this runbook must mirror exactly.

From `scripts/upload-plates.js` lines 30-32 + 96-103:
```javascript
const BUNNY_STORAGE_HOST = process.env.BUNNY_STORAGE_HOST ?? 'la.storage.bunnycdn.com';
const BUNNY_ZONE = process.env.BUNNY_ZONE ?? 'pnwmoths';
const BUNNY_API_KEY = process.env.BUNNY_API_KEY ?? '';

const args = [
  '-s', '-S', '-f',
  '-X', 'PUT',
  '-H', `AccessKey: ${BUNNY_API_KEY}`,
  '-H', 'Content-Type: application/octet-stream',
  '--data-binary', `@${localPath}`,
  `https://${BUNNY_STORAGE_HOST}/${BUNNY_ZONE}/${cdnPath}`,
];
```

Storage path convention (from RESEARCH.md § URL Convention Confirmation):
- Storage host: `la.storage.bunnycdn.com` (zone is `pnwmoths`)
- Storage path: `species-tiles/{slug}/{specimen_id}-{view}/{file}` — NO leading slash, NO `pnwmoths/` prefix in the path (zone name is the zone root)
- Pull Zone URL: `https://pnwmoths.b-cdn.net/species-tiles/{slug}/{specimen_id}-{view}/{file}`

CDN verification commands (post-upload):
- `curl -I "https://pnwmoths.b-cdn.net/species-tiles/{slug}/{specimen_id}-{view}/{specimen_id}-{view}.dzi"` → expect 200
- `curl -I "https://pnwmoths.b-cdn.net/species-tiles/{slug}/{specimen_id}-{view}/{specimen_id}-{view}_files/0/0_0.jpg"` → expect 200
- `curl -I -H "Origin: https://pnwmoths.b-cdn.net" "https://pnwmoths.b-cdn.net/species-tiles/.../{prefix}.dzi"` → check `access-control-allow-origin` response header (RESEARCH.md Pitfall 3)

Security (from RESEARCH.md § Security Domain):
- BUNNY_API_KEY MUST be in environment, never logged, never committed; the recipe MUST instruct the operator to NOT paste their key into any committed file or chat output

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator host → bunny.net Storage (HTTPS PUT, AccessKey header) | Auth via `BUNNY_API_KEY` env var; same auth model as Phase 13 / Phase 18 |
| bunny.net Storage → Pull Zone → public CDN | Public-read; tiles are public moth specimen images; no access control |
| Browser → bunny.net Pull Zone (XHR for .dzi descriptor) | CORS-gated for XHR; tile `<img>` loads are CORS-exempt |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-03-01 | Information Disclosure | BUNNY_API_KEY in operator shell history / logs | mitigate | UPLOAD-RECIPE.md `## Prerequisites` + closing security note instruct operator to never commit / never paste into chat / never echo to logs; mirrors `scripts/ingest-photos.js` redact() pattern from Phase 26 |
| T-28-03-02 | Tampering | Wrong storage path layout (leading slash, zone prefix) → tiles unreachable | mitigate | UPLOAD-RECIPE.md `## When Things Go Wrong` covers (b) and (c); `## Verify (CDN reachability)` block requires `curl -I` 200 on every pair before plan completes |
| T-28-03-03 | Denial of Service | CORS misconfiguration on Pull Zone blocks .dzi XHR → OSD blank canvas in Plan 04 | mitigate | Explicit CORS verification step (Task 2 verify step 4) records header state BEFORE Plan 04 wires the browser; if absent, surfaces the need for Pull Zone CORS rule before Plan 05's browser verification |
| T-28-03-04 | Tampering | Malicious tile content uploaded to wrong species path | accept | Operator is the only writer (single BUNNY_API_KEY holder); same trust model as plates + images zones |

Block-severity threshold: high. T-28-03-03 (CORS) is the highest-priority risk because it would block the browser verification milestone — mitigated by making CORS the explicit closing check of this plan, before Plan 04 depends on it.
