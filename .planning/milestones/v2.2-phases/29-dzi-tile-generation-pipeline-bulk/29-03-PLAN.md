    Create `_instructions/TILING_HIGH_RES_PHOTOS.md` matching the section structure and tone of `_instructions/INGESTING_HIGH_RES_PHOTOS.md`. Required sections, in this order:

    **1. Overview (1-2 paragraphs)**
    - The script reads `data/species-photos-manifest.csv` and processes each eligible row (`match_bucket` ∈ {clean-match, slug-match, resolved-via-synonym}, `status` not yet `tiled`).
    - For each row: download the source TIFF from Dropbox into the TIFF cache (if not already cached), then run `vips dzsave` to produce a DZI tile pyramid into the configured output directory, then advance the manifest row's `status` to `tiled`.
    - Idempotent: rerunning skips rows whose `status` is already `tiled` AND whose `.dzi` already exists on disk.

    **2. Prerequisites**
    - Datacenter server access (SSH).
    - `libvips` CLI installed — confirm with `vips --version`. Installation: `sudo apt install libvips-tools` (Debian/Ubuntu).
    - Node.js version per `.nvmrc`.
    - `data/species-photos-manifest.csv` checked into the repo and pulled to the server (the manifest from Phase 26 + Phase 27 curation).
    - Outbound HTTPS access to `content.dropboxapi.com` and `api.dropboxapi.com`.
    - A Dropbox app access token with `files.metadata.read` + `sharing.read` scopes (generate at https://www.dropbox.com/developers/apps).
    - Adequate disk on the datacenter server: ~9 GB headroom for tile output (PILOT-LESSONS.md projection: ~850 KB × 10,000 pairs ≈ 8.5 GB). TIFF cache size scales with corpus footprint (~204 GB if every TIFF is downloaded; less if the operator deletes cached TIFFs incrementally).

    **3. Configuration**
    - The committed file `scripts/tile-config.json` carries the seven parameters: `tileSize` (256), `overlap` (1), `suffix` (`.webp[Q=80]`), `layout` (`dz`), `tileOutputDir`, `tiffCacheDir`, `dropboxShareUrl`.
    - Two env vars override the path keys without editing the committed config: `TILE_OUTPUT_DIR` and `TIFF_CACHE_DIR`. Use these to point at server-local volumes without making per-host commits.
    - `DROPBOX_TOKEN` is REQUIRED for any non-dry-run invocation. Never commit, log, or paste the token into shared chat. Treat it as a secret.

    **4. Run the dry-run preview**
    - Show the literal command: `DRY_RUN=1 npm run photos:tile`
    - Expected output: a list of the first 5 eligible rows with their computed tile prefixes and source paths. No manifest write happens. No Dropbox calls happen unless `DROPBOX_TOKEN` is set; if absent, the script prints a structure-validation message and exits.
    - Operator checklist before proceeding to the real run: confirm output paths look right; confirm the count of eligible rows is sane (should be roughly the count of `clean-match` + `slug-match` + `resolved-via-synonym` rows in the manifest, modulo rows already at `status: tiled`).

    **5. Run the full pipeline**
    - Show the literal command: `DROPBOX_TOKEN=sl.xxxxx npm run photos:tile 2>&1 | tee tile-run-$(date +%F-%H%M).log`
    - Document the log format: one line per row transition with ISO timestamp, 12-char content_hash prefix, action (download/tile), outcome (ok / cache-hit / already-on-disk-advance / failed). Mirror the format from `_instructions/INGESTING_HIGH_RES_PHOTOS.md` §"Output you'll see".
    - Expected runtime: ~10,000 pairs × (download + vips dzsave). PILOT-LESSONS.md observed vips dzsave at ~1–3 seconds per pair on pilot TIFFs; the dominant cost is Dropbox download bandwidth.

    **6. Resume after interruption**
    - kill -9 / SIGINT / network drop / power loss are all recoverable.
    - On the next `npm run photos:tile`, three idempotency checks gate work:
      1. Rows with `status: tiled` are skipped silently.
      2. Rows whose `.dzi` already exists on disk get their `status` advanced to `tiled` without re-running vips (this catches the kill-between-vips-and-manifest-write window).
      3. Rows whose TIFF is already cached locally skip the download step.
    - Manifest is checkpoint-written every 25 rows; worst-case loss is 24 rows of in-memory progress.

    **7. When things go wrong**
    Mirror the §"When Things Go Wrong" pattern from `TILE-RECIPE.md`. Required entries (one paragraph each):
    - `vips: command not found` — `libvips` CLI is not installed; `sudo apt install libvips-tools`.
    - `vips dzsave` writes `.png` or `.jpg` tiles instead of `.webp` — `scripts/tile-config.json` `suffix` was edited. Restore to `.webp[Q=80]` and rerun (existing tiles get overwritten on re-run only if the row is not yet `status: tiled` — if it is, the operator must delete the `.dzi` and `_files/` for that prefix before rerunning).
    - Dropbox download fails with 401 — `DROPBOX_TOKEN` expired or has the wrong scopes. Regenerate at https://www.dropbox.com/developers/apps with `files.metadata.read` + `sharing.read`.
    - Dropbox download fails with 429 — withRetry will back off (2/4/8/16/32 s). If the run still ultimately fails, the row is marked `status: failed` with the last_error and the run continues; rerun later.
    - `vips dzsave` fails on a specific TIFF — vips error is captured into `last_error`; the row is `status: failed`. Inspect manually; if the file is corrupt, re-download it from Dropbox via the web UI and replace the cached file at `{TIFF_CACHE_DIR}/{content_hash}-{filename_raw}`; the row's next run will pick up the new file.
    - Disk full in `tileOutputDir` or `tiffCacheDir` — free space, then rerun; idempotency means no work is lost.
    - Mixed-case directory appears in the tile output — should never happen because the script lowercases `species_slug` before path joining. If it does, file a bug — Plan 02's lowercasing logic was bypassed.

    **8. Verification**
    - Spot-check three random `tiled` rows: confirm the `.dzi` descriptor exists at `{tileOutputDir}/{slug}/{specimen_id}-{view}.dzi`, the `_files/` tile pyramid sibling exists, and the descriptor's `Format` attribute is `webp`.
    - Confirm `data/species-photos-manifest.csv` is updated in-place (status transitions visible in `git diff`).
    - Save the run log alongside the commit message for Phase 30's storage-footprint sanity check.

    **9. Next phase handoff**
    - Phase 30 (`UPLOAD-01`) consumes manifest rows with `status: tiled` and uploads the tile directories to bunny.net. Do not delete tiles or TIFFs until Phase 30 has shipped successfully.
