```js
function substituteTerms(textNode, terms, seen) {
  const rawText = textNode.rawText; // rawText preserves existing HTML entities

  // Find the positionally-earliest unseen term match in this text node.
  // We cannot break on the first longest-first term: a longer term appearing
  // later in the text would push a positionally-earlier shorter term into the
  // "before" fragment, which is never re-visited by the outer loop.
  let earliest = null; // { entry, match }
  for (const entry of terms) {
    if (seen.has(entry.lower)) continue;

    entry.regex.lastIndex = 0; // reset stateful gi regex before each exec
    const match = entry.regex.exec(rawText);
    if (!match) continue;

    if (earliest === null || match.index < earliest.match.index) {
      earliest = { entry, match };
    }
  }

  if (!earliest) return;

  const { entry, match } = earliest;
  seen.add(entry.lower);

  const before = rawText.slice(0, match.index);
  const matched = match[0]; // preserves original case from source text
  const after = rawText.slice(match.index + matched.length);

  const abbr =
    `<abbr class="glossary-term" ` +
    `title="${escapeHtml(entry.definition)}" ` +
    `data-definition="${escapeHtml(entry.definition)}" ` +
    `data-image-url="${escapeHtml(entry.imageUrl)}"` +
    `>${matched}</abbr>`;

  textNode.parentNode.exchangeChild(textNode, parse(before + abbr + after));
}
```

Do NOT change: function signature, JSDoc comment, `escapeHtml`, `escapeRegex`, `buildTermMap`,
or `applyGlossaryTerms`. Only `substituteTerms` changes.

Why this fix works:
- The longest-first sort in `buildTermMap` prevents "wing" from matching inside "forewing"
  because "forewing" appears earlier in the terms array AND earlier in the text, so it will
  always have a lower (or equal) match.index — the min-index selection picks it first.
- When two truly distinct terms appear in the same text node (e.g., "forewing" at index 41
  and "outer margin" at index 76), the min-index scan correctly selects "forewing" (index 41).
- The outer `applyGlossaryTerms` loop re-processes the same element's text nodes after each
  substitution, so "outer margin" will be wrapped on the next pass through that element.
