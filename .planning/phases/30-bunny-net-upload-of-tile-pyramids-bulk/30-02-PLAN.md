    ### Section 6 — `## Resume After Interruption`
    Three subsections covering the recovery semantics (D-02 — whole-directory granularity):

    `### What's recoverable`
    Bulleted: kill -9 / Ctrl-C / SIGINT, network drop, transient bunny.net 5xx, laptop sleep.

    `### What re-running does`
    On the next `npm run photos:upload`:
    1. `isUploadable` filter excludes rows at `status: uploaded` — they produce no log output
    2. Rows that crashed mid-directory stayed at `status: tiled` — the script re-uploads the ENTIRE directory (bunny.net PUT is idempotent; already-uploaded files are overwritten safely)
    3. Manifest is checkpoint-written every 25 rows; a kill -9 loses at most 24 rows of in-memory status transitions

    `### What's NOT recoverable`
    A row whose tiles were deleted (status advance succeeded) but whose status row was lost in the 24-row checkpoint window. On restart, the row is `status: tiled` (in-memory advance was not flushed) but the tile directory is gone. The script tries to upload a missing directory and fails the row gracefully (ENOENT → caught → `status: failed`). Operator remediation: re-tile the affected rows with `npm run photos:tile` (Phase 29), then re-run `npm run photos:upload`. This is the documented tradeoff (D-02/D-03, RESEARCH.md Pitfall 1).

    Subsection `### Do NOT edit the manifest while the script is running.` The script holds the full manifest in memory and writes it periodically; mid-run edits will be overwritten. Same as the Phase 29 runbook.

    ### Section 7 — `## When Things Go Wrong`
    Troubleshooting entries (bold question/symptom, then explanation and remediation), at least 5 entries. Source these from RESEARCH.md "Common Pitfalls" 1-6:

    **`BUNNY_API_KEY is required`** — Set the env var on the invocation line. Confirm with `bunny.net dashboard → Storage → pnwmoths → FTP & API Access → Password`.

    **Every PUT fails immediately with HTTP 4xx** — Two common causes: (a) BUNNY_API_KEY is wrong; verify against the dashboard. (b) The PUT is going to the wrong host. Confirm `BUNNY_STORAGE_HOST` is `la.storage.bunnycdn.com` (Storage Zone), NOT `pnwmoths.b-cdn.net` (Pull Zone — read-only). The script's `withRetry` wastes 62 seconds (2+4+8+16+32) before the first row fails on a bad API key — `Ctrl-C` after the first failure and fix the env var.

    **Mixed-case slug appears in CDN path** — This should never happen — the script calls `.toLowerCase()` on `species_slug` unconditionally. If you see an uppercase slug in DRY_RUN output, file a bug against Plan 01.

    **`ENOENT` errors on the tile directory** — A row reached `status: tiled` in memory, tile deletion succeeded, but the manifest checkpoint flush didn't happen before a kill -9 (the 24-row window). Re-tile those rows with `npm run photos:tile` (Phase 29) and re-run the upload.

    **Pre-flight walk takes more than 90 seconds** — Expected on a corpus with hundreds of thousands of files. The script prints `[upload-tiles] Pre-flight: measuring tile corpus size (this may take 30-90s)...` before starting the walk so you know to wait. If the walk hangs past 5 minutes, something is wrong (filesystem corruption, slow external disk) — Ctrl-C and investigate.

    **Disk fills up mid-run** — Should not happen given D-03 (per-row deletion shrinks disk usage monotonically as the run proceeds). If it does, free space manually and re-run; idempotency preserves all already-uploaded rows.
