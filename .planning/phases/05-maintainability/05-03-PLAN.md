<task type="auto">
  <name>Task 1: Bake npm ci into Dockerfile and remove shadowing node_modules volume</name>
  <files>Dockerfile, docker-compose.yml</files>
  <action>
In Dockerfile, after the `WORKDIR /workspace` line, add:

```
COPY package*.json ./
RUN npm ci
```

This bakes node_modules into the image layer so every container — including cold starts — has packages available immediately. No entrypoint script is needed.

In docker-compose.yml, remove the named `node_modules` volume mount from the `dev` service volumes list (delete the `- node_modules:/workspace/node_modules` line) and remove the top-level `volumes: node_modules:` declaration entirely. The volume was intended to cache node_modules across container restarts, but it starts empty on cold start and silently shadows the image's node_modules — which is the root cause of the ERR_MODULE_NOT_FOUND failure. With npm ci in the image, the volume provides no benefit.

After editing, the Dockerfile runtime stage should end:
```
WORKDIR /workspace
COPY package*.json ./
RUN npm ci
```

And docker-compose.yml volumes for the dev service should only contain:
```
volumes:
  - .:/workspace
```
with no top-level `volumes:` block.
  </action>
  <verify>
Run: docker compose build dev 2>&1 | tail -5
Expected: build succeeds (exit 0), no errors about missing packages.

Then cold-start test: docker volume rm pnwmoths_node_modules 2>/dev/null || true && docker compose run --rm dev node -e "require('@duckdb/node-api')" 2>&1
Expected: exits 0 (no ERR_MODULE_NOT_FOUND).
  </verify>
  <done>
`docker compose run --rm dev npm run build` completes successfully from a clean state with no ERR_MODULE_NOT_FOUND for @duckdb/node-api or any other package. The named node_modules volume no longer exists in docker-compose.yml.
  </done>
</task>
