1. Create `eleventy.config.js` as ESM (`export default function`):
   ```javascript
   export default function (eleventyConfig) {
     // Passthrough copy: per-species Parquet files from data/parquet/{slug}/ to species/{slug}/
     // This maps data/parquet/acronicta-americana/records.parquet -> _site/species/acronicta-americana/records.parquet
     eleventyConfig.addPassthroughCopy({
       "data/parquet/**/*.parquet": "species/"
     });

     return {
       dir: {
         input: "src",
         output: "_site",
         data: "_data"
       }
     };
   }
   ```
   NOTE: The passthrough copy glob `data/parquet/**/*.parquet` needs to map correctly. The source structure is `data/parquet/{slug}/records.parquet` and the target should be `_site/species/{slug}/records.parquet`. Eleventy's addPassthroughCopy with object form strips the key prefix and adds the value prefix. So `{ "data/parquet": "species" }` maps `data/parquet/acronicta-americana/records.parquet` to `_site/species/acronicta-americana/records.parquet`. Use this simpler form: `eleventyConfig.addPassthroughCopy({ "data/parquet": "species" })`.

2. Create `src/_data/species.js` as an async default export (per RESEARCH.md Pattern 2):
   ```javascript
   import { DuckDBInstance } from '@duckdb/node-api';

   export default async function () {
     const db = await DuckDBInstance.create(':memory:');
     const conn = await db.connect();

     await conn.run(`
       CREATE TABLE species AS
       SELECT * FROM read_csv('data/species.csv',
         header = true,
         columns = {
           'id': 'INTEGER',
           'genus': 'VARCHAR',
           'species': 'VARCHAR',
           'common_name': 'VARCHAR',
           'noc_id': 'VARCHAR',
           'authority': 'VARCHAR'
         }
       )
     `);

     const result = await conn.runAndReadAll(`
       SELECT
         id,
         genus,
         species,
         common_name,
         noc_id,
         authority,
         lower(genus || '-' || species) AS slug
       FROM species
       ORDER BY genus, species
     `);

     await conn.close();
     return result.rows;
   }
   ```

   IMPORTANT: The `runAndReadAll` method returns a result object. Check the actual shape at runtime -- it may be `result.rows` (array of objects) or `result.getRows()` (method). The RESEARCH.md assumes `result.rows` based on blog post examples. If that does not work, try `result.getRows()` or iterate using `result.currentRowCount` and column accessors. Log the result object to inspect its shape if the first attempt fails.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CSV data -> Eleventy data file | Species names from CSV flow through DuckDB into Eleventy templates |
| Parquet passthrough copy | Files from data/parquet/ are copied verbatim to _site/ |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Tampering | Species names in HTML output | accept | Species names are rendered by Nunjucks auto-escaping (default in Eleventy 3.x); HTML injection via genus/species names is mitigated by template engine |
| T-01-06 | Information Disclosure | Parquet passthrough copy | accept | Parquet files contain occurrence data that is intentionally public (lat/long, dates, collectors). No sensitive data in the pipeline. |
| T-01-07 | Tampering | Slug used in permalink path | mitigate | Slug is generated via DuckDB `lower(genus \|\| '-' \|\| species)` which only concatenates two VARCHAR columns. Plan 01 validates that genus/species contain only expected characters. Eleventy permalink uses the slug directly; no path traversal possible since Eleventy normalizes output paths within dir.output. |
</threat_model>
