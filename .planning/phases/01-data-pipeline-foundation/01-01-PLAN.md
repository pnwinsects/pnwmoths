<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CSV input -> build script | Untrusted CSV files (edited by non-technical maintainers, possibly via Excel) enter the build pipeline |
| DuckDB SQL construction | Species names from CSV used in file paths and SQL strings |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | CSV input files | mitigate | Pre-flight UTF-8 validation in validateCsv(); DuckDB strict mode rejects malformed rows; explicit column typing prevents type confusion |
| T-01-02 | Tampering | Slug-based file paths | mitigate | Slug generated from genus+species via toLowerCase().replace(); validate that genus and species columns contain only alphanumeric characters and spaces before slug generation |
| T-01-03 | Tampering | SQL injection via CSV values | accept | DuckDB read_csv parses CSV into typed columns; values never interpolated into SQL strings (species_id is INTEGER from typed column). The only interpolated value in COPY TO is the slug (file path), which is validated. |
| T-01-04 | Denial of Service | Extremely large CSV input | accept | Build-time only; no runtime exposure. DuckDB handles large CSV efficiently. Maintainer controls input files. |
</threat_model>
