    - Capitalized second token (e.g. 'Lasionycta Carolynae-A-D.tif') → { binomial: null, bucketHint: null } (no coercion — REPORT.md unparseable case)

  <action>
    Port `extractBinomial` from spike `parse-classify.mjs:110-127` into `scripts/lib/parse-photo-filename.js`. Apply the three D-14 fixes called out in 26-PATTERNS.md:

    FIX #1 — replace the spike's `b.length >= 3` guard with `b.length >= 2` so `Trichoplusia ni` and `Rachiplusia ou` parse cleanly.

    FIX #2 — replace the second-token regex `/^[a-z]+$/` with `/^[a-z]+(-[a-z]+)?$/` so `Xestia c-nigrum` and `Autographa v-alba` parse.

    FIX #3 — before the per-pair walk, scan the token stream for provisional markers (`sp`, `nr`, or the consecutive pair `n` followed by `sp`). On hit return `{ binomial: null, bucketHint: 'provisional' }`. Do this BEFORE the clean-binomial walk so `Monostoecha n sp` does not accidentally match `Monostoecha n` as Genus + 2-char epithet.

    Return shape is `{ binomial, bucketHint }` for every call — never raw strings, never null. (The spike returned `string | null`; this library is richer because Phase 03 needs to distinguish "no binomial" from "no binomial, provisional bucket".)

    Add `parseSpecimenAndView(fname)` — adapt `scripts/migrate-images.js:91-98` but loosen the regex from `/-([A-Z])-([A-Z])\.[^.]+$/` to `/-([A-Z0-9_]+)-([DV])\.[^.]+$/` so institutional IDs (`OSAC_0001081322`, `WWUC000000083`) are admitted. View output is the raw single-letter `D` / `V` — NOT translated to 'dorsal'/'ventral' (the manifest column is the raw letter per D-05).

    Add `toSpeciesSlug(binomial)` — for matching against `data/species.csv` slug map. Mirror the spike's `slug = ${genus}-${species}.toLowerCase()` rule (parse-classify.mjs:95). One-liner: `binomial.toLowerCase().trim().replace(/\s+/g, '-')`.

    Module shape: `export function ...` for each helper, no default export, ESM (project is `"type": "module"`). No I/O, no node:fs imports — this is pure logic so it loads in tests in <10ms.

| filename → parser | Untrusted Dropbox filenames cross into the parser; the parser is pure so the threat surface is regex DoS and incorrect classification, NOT code execution. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26.01-01 | Tampering | extractBinomial / parseSpecimenAndView regex patterns | mitigate | All regexes are bounded (no nested quantifiers, no catastrophic backtracking). The longest input is a ~120-char filename; regex execution is O(n). Verified by inspection of the spike regex `/[_\-\.]+/g` (single char class, +) and `/^[a-z]+(-[a-z]+)?$/` (no nesting). |
| T-26.01-02 | Information disclosure | parser error paths | accept | Parser returns null on bad input; never throws; never logs. No secrets present in this layer (DROPBOX_TOKEN never enters the pure helper). |
| T-26.01-03 | Repudiation | classification correctness for provisional IDs | mitigate | FIX #3 routes `n sp`/`sp`/`nr` to provisional with binomial null. Unit tests pin this so a future refactor cannot silently promote provisional rows to clean-match. |
</threat_model>
