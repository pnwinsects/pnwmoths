2. Name: `pnwmoths` (this becomes the FTP username and the path prefix in rclone)

4. Click Create

5. Navigate to CDN → Add Pull Zone

9. In the Storage Zone details, locate the FTP credentials:
   - FTP host: `ny.storage.bunnycdn.com` (or regional equivalent shown in dashboard)

  <how-to-verify>
**Step 1: Install rclone if not installed:**
```bash
rclone version
# If not installed: brew install rclone  (macOS)
```

**Step 2: Configure rclone FTP remote for bunny.net:**
```bash
rclone config
```
- Select `n` for new remote
- Name: `bunny`
- Type: `ftp`
- Host: `ny.storage.bunnycdn.com` (use the regional endpoint shown in your Storage Zone FTP credentials)
- User: `pnwmoths` (your Storage Zone name)
- Password: run `rclone obscure YOUR_FTP_PASSWORD` and paste the output
- Port: `21` (default)
- Leave all other settings at defaults
- Save

**Step 3: Verify rclone connection:**
```bash
rclone ls bunny:pnwmoths/
# Expected: empty listing (no files yet). An error here means credentials are wrong.
```

**Step 4: Dry-run to preview what will be uploaded:**
```bash
DRY_RUN=1 node scripts/migrate-images.js 2>&1 | head -40
# Review: should show rclone copy commands for moths/ and glossary-images/
```

**Step 5: Run the actual upload:**

If the Django DB export (data/django-photos.csv) is available, ensure it is in place before running.
If not yet available, the script proceeds with empty photographer/license fields — acceptable for now.

```bash
node scripts/migrate-images.js
# This uploads ~4,577 species photos and ~15 glossary images via rclone copy --ignore-times
# Expect significant transfer time (~hours for full upload over FTP)
# Progress is shown via rclone's stdio: inherit output
```

**Step 6: Verify upload completed:**
```bash
rclone ls bunny:pnwmoths/ | wc -l
# Expected: number of directories (one per species slug + glossary/)

rclone ls bunny:pnwmoths/acronicta-americana/ 2>/dev/null | head -5
# Expected: lists the Acronicta americana files
```

**Step 7: Spot-check CDN delivery:**
```bash
# Pick any uploaded file and verify CDN serves it
curl -I "https://pnwmoths.b-cdn.net/acronicta-americana/Acronicta%20americana-A-D.jpg"
# Note: URL-encode spaces as %20
# Expected: HTTP 200, Content-Type: image/webp (with Optimizer active)
```

NOTE: Do NOT use `rclone sync` at any point. `rclone sync` deletes files in the destination that are absent locally — this would empty the bucket. Always use `rclone copy`.
  </how-to-verify>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer → bunny.net dashboard | Account credentials; dashboard actions affect production CDN |
| rclone → bunny.net FTP | FTP password stored in ~/.config/rclone/rclone.conf (local machine only) |
| CDN Pull Zone → public internet | All CDN content is publicly accessible by design |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-03-01 | Tampering | rclone sync accidentally emptying CDN bucket | mitigate | PITFALLS.md Pitfall 3 documented; Task 3 explicitly forbids rclone sync; migrate-images.js uses copy exclusively |
| T-13-03-02 | Information Disclosure | FTP password visible in rclone.conf | accept | rclone uses obscured (not plaintext) password storage; file is on developer's local machine, not in repo |
| T-13-03-03| Tampering | Confusing Storage Zone URL with Pull Zone URL | mitigate | Task 1 step 11 explicitly verifies Pull Zone hostname; PITFALLS.md Pitfall 5 documented; CDN_BASE_URL hardcoded as Pull Zone URL |
| T-13-03-04 | Denial of Service | Optimizer not enabled — templates use resize params that are silently ignored | mitigate | Task 2 requires browser verification of WebP + resize before marking complete; Pitfall 7 documented |
| T-13-03-05 | Tampering | Stale CDN cache after re-upload of same filename | accept | Documented in PITFALLS.md Pitfall 6; initial migration uses new filenames, so no cache invalidation needed for first upload |
</threat_model>
